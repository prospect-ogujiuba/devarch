**compose/stack.go — loadEffectiveEnvVars:**
The compose generator currently loads env vars as plaintext `map[string]string` for YAML output. This is the compose preview endpoint and actual compose generation — secrets must not appear in plaintext.

Two-layer redaction (per user decision: belt and suspenders):
1. When loading from `service_env_vars` and `instance_env_vars`, check `is_secret` flag. If `is_secret=true`, replace value with empty string in the merged map (compose needs the KEY present for the container but the value should be redacted). Actually, compose generation needs the REAL value for the container to work — so compose generation for actual `apply` needs plaintext, but compose PREVIEW needs redaction.

Approach: Add a `redactSecrets bool` parameter to `GenerateStack`. When `redactSecrets=true` (preview/compose endpoint), replace secret values with `"***"` in the environment map. When `false` (apply flow), use real values.

Modify `loadEffectiveEnvVars` to also SELECT `is_secret` from both `service_env_vars` and `instance_env_vars`. Track which keys are secret. After the three-layer merge (template + wired + instance), if `redactSecrets` is true, replace values for secret keys with `"***"`.

Alternatively (simpler): the `loadEffectiveEnvVars` can return both the merged map AND a set of secret keys. Then in `GenerateStack`, if redact mode, mask those keys.

The compose preview handler (`stack_compose.go`) should call with `redactSecrets=true`. The apply handler (`stack_apply.go`) should call with `redactSecrets=false` (or the apply flow that calls compose up needs real values).

**stack_compose.go:**
- Pass `redactSecrets=true` when calling `GenerateStack` for the preview/download endpoint.

**stack_plan.go:**
- The Plan endpoint returns `WirePlanEntry` with `InjectedEnvVars` map. These may contain sensitive values. If any injected env var key matches `export.IsSecretKey()`, redact its value to `"***"`.
- The plan output should NOT contain plaintext secrets in any field.

**stack_export.go / export/exporter.go:**
- The exporter loads env vars for each instance. Currently `loadInstanceDef` queries `instance_env_vars` and `service_env_vars`. The existing `export/secrets.go:RedactSecrets` already applies keyword heuristic.
- Additionally: check `is_secret` flag from DB. If `is_secret=true`, redact regardless of keyword match. This ensures DB-flagged secrets are always redacted in export, even if key name doesn't match heuristic.
- Modify the exporter's env var loading to also SELECT `is_secret`. For any row with `is_secret=true`, set value to `${SECRET:KEY_NAME}` placeholder (matching existing export format). Then also apply `RedactSecrets()` for keyword heuristic (belt and suspenders).
  </action>

**instance_overrides.go — Resource limits endpoint:**
Add `UpdateResourceLimits` handler (PUT) and `GetResourceLimits` handler (GET) to InstanceHandler.

Request body for PUT:
```json
{
  "cpu_limit": "2.0",
  "cpu_reservation": "0.5",
  "memory_limit": "1g",
  "memory_reservation": "512m"
}
```
All fields optional (nullable). Use UPSERT pattern: `INSERT INTO instance_resource_limits ... ON CONFLICT (instance_id) DO UPDATE SET ...`. If all fields are null/empty, DELETE the row instead (clean up — no deploy section if no limits).

GET returns the current resource limits or 404 if none set.

Add validation warnings (not errors — per user decision, never block):
- memory_limit parsed: warn if < 4MB ("memory limit very low, container may fail to start")
- cpu_limit parsed: warn if < 0.01 ("CPU limit extremely low")
- Return warnings in response alongside saved data.

Parse memory strings (4m, 512m, 1g, 2G) — case-insensitive suffix matching.

**routes.go:**
Register routes under the instance route:
```
r.Get("/resources", instanceHandler.GetResourceLimits)
r.Put("/resources", instanceHandler.UpdateResourceLimits)
```

**compose/stack.go — Deploy resources section:**
Add `deployConfig`, `resourcesConfig`, `resourceLimits` types to `stackServiceEntry`:
```go
type stackServiceEntry struct {
    // ... existing fields ...
    Deploy *deployConfig `yaml:"deploy,omitempty"`
}
```

In `GenerateStack`, after loading instance effective config, query `instance_resource_limits` for each instance. If limits exist, populate the `Deploy` field on the service entry.

Add `loadResourceLimits(instancePK int) (*deployConfig, error)` method to Generator. Query `instance_resource_limits` table. If no row or all fields null, return nil (omit deploy section). Otherwise build the deploy config with limits and/or reservations as applicable.

**plan/types.go — Resource limits in plan:**
Add `ResourceLimits` field to plan types so limits appear in plan diff output. Add a `ResourceLimitEntry` type:

```go
type ResourceLimitEntry struct {

    CPUReservation    string `json:"cpu_reservation,omitempty"`
    MemoryLimit       string `json:"memory_limit,omitempty"`
    MemoryReservation string `json:"memory_reservation,omitempty"`
}
```

**stack_plan.go:**
In the Plan handler, load resource limits for each instance in the plan response. Add warnings for instances with unreasonable limits. Add resource limits to the plan response so the dashboard can display them.
