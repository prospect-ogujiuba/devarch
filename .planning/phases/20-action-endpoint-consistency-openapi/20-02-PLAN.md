**Step 4: Annotate every handler function.**

For each handler, add swaggo comment annotations directly above the function. Cross-reference routes.go for exact paths. Use chi `{param}` path syntax (NOT `:param`).

Annotation rules:
- `@Summary` — imperative, one-line (e.g., "Start a stack")
- `@Description` — additional context if needed
- `@Tags` — single resource tag: `stacks`, `instances`, `services`, `categories`, `projects`, `networks`, `nginx`, `runtime`, `status`, `registries`, `auth`, `proxy`, `websocket`
- `@Accept json` and `@Produce json` for JSON endpoints
- `@Param` — path params (`path`), query params (`query`), body params (`body`) with type
- `@Success` — use nested envelope syntax: `{object} respond.SuccessEnvelope{data=TYPE}` for data responses, `{object} respond.SuccessEnvelope{data=respond.ActionResponse}` for action endpoints
- `@Failure` — use `{object} respond.ErrorEnvelope` for all error codes
- `@Router /path [method]` — exact path from routes.go with `/api/v1` prefix stripped (swag uses BasePath)
- `@Security ApiKeyAuth` on all endpoints inside the `r.Route("/api/v1", ...)` group
- NO `@Security` on auth/validate and auth/ws-token (registered outside auth middleware group)

**Example patterns by endpoint type:**

Action endpoint (start/stop/restart/apply/resolve):
```go
// Start godoc
// @Summary      Start a stack
// @Description  Start all instances in a stack. Stack must be enabled.
// @Tags         stacks
// @Produce      json
// @Param        name path string true "Stack name"
// @Success      200 {object} respond.SuccessEnvelope{data=respond.ActionResponse}
// @Failure      404 {object} respond.ErrorEnvelope
// @Failure      409 {object} respond.ErrorEnvelope
// @Failure      500 {object} respond.ErrorEnvelope
// @Router       /stacks/{name}/start [post]
// @Security     ApiKeyAuth
```

  <action>
Create `.github/workflows/openapi-check.yml`:

```yaml
name: OpenAPI Validation

on:
  pull_request:
    paths:
      - 'api/internal/api/**'
      - 'api/cmd/server/main.go'
      - 'api/docs/**'

jobs:
  openapi-diff:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Set up Go
        uses: actions/setup-go@v5
        with:
          go-version: '1.22'

      - name: Install tools
        run: |
          go install github.com/swaggo/swag/cmd/swag@latest
          go install github.com/oasdiff/oasdiff@latest

      - name: Generate PR spec
        working-directory: api
        run: swag init -g cmd/server/main.go --output docs --parseDependency --parseInternal

      - name: Save PR spec
        run: cp api/docs/swagger.yaml /tmp/pr-swagger.yaml

      - name: Generate base spec
        run: |
          git checkout ${{ github.event.pull_request.base.sha }}
          cd api
          swag init -g cmd/server/main.go --output docs --parseDependency --parseInternal || true
          cp docs/swagger.yaml /tmp/base-swagger.yaml 2>/dev/null || echo '{"openapi":"3.0.0","info":{"title":"DevArch API","version":"1.0"},"paths":{}}' > /tmp/base-swagger.yaml

      - name: Restore PR branch
        run: git checkout ${{ github.event.pull_request.head.sha }}

      - name: Check for breaking changes
        run: |
          oasdiff breaking /tmp/base-swagger.yaml /tmp/pr-swagger.yaml \
            --fail-on ERR \
            --format githubactions || true

      - name: Generate changelog
        if: always()
        run: |
          oasdiff changelog /tmp/base-swagger.yaml /tmp/pr-swagger.yaml \
            --format markdown >> $GITHUB_STEP_SUMMARY 2>/dev/null || echo "No changes detected" >> $GITHUB_STEP_SUMMARY
```

Note: The `|| true` on base spec generation handles the case where the base branch doesn't have swag annotations yet (first PR to add them). The fallback creates a minimal empty spec so oasdiff can compare. The `--fail-on ERR` flag means only truly breaking changes (removed endpoints, changed required params) fail the check, not warnings.
  </action>
