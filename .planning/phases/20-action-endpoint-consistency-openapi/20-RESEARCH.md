# Phase 20: Action Endpoint Consistency & OpenAPI - Research

**Researched:** 2026-02-11
**Domain:** API response standardization and OpenAPI specification generation
**Confidence:** HIGH

## Summary

Phase 20 addresses two interconnected requirements: standardizing action endpoint responses and generating OpenAPI documentation. Current action endpoints (start/stop/restart/apply/resolve) return inconsistent fields—some return only `status`, others include `output`, `warnings`, or `lock_warnings`. The codebase uses Go 1.22 with chi router and has minimal dependencies, making it ideal for annotation-based OpenAPI generation via swaggo/swag.

The standardization challenge is determining what fields action responses should include. Container lifecycle operations typically don't provide detailed metadata (container client methods return only errors), but some actions (apply, resolve) already return richer data. The solution requires defining a consistent response structure that accommodates both simple operations (start/stop) and complex operations (apply with lock warnings).

For OpenAPI generation, swaggo/swag is the industry standard for Go with 18k+ GitHub stars and active maintenance through 2025-2026. It generates OpenAPI 3.0 specs from code annotations, integrates seamlessly with chi router, and supports CI validation via diff detection tools (oasdiff, openapi-changes). The annotation-based approach (code-first) fits DevArch's pattern of minimal abstraction and avoids the coupling of spec-first approaches like oapi-codegen.

**Primary recommendation:** Define a common `ActionResponse` struct with optional fields, annotate handlers with swaggo comments, generate spec with `swag init`, and validate spec changes in CI using oasdiff with `--fail-on-incompatible`.

### Pattern 1: Action Response Standardization

**What:** Define consistent response structure for all action endpoints (start/stop/restart/apply/resolve)

**When to use:** All POST endpoints that trigger container operations

**Current inconsistencies:**
- Stack lifecycle: `{"status": "started|stopped|restarted"}`
- Stack apply: `{"status": "applied", "output": string, "lock_warnings": []}`
- Wiring resolve: `{"resolved": int, "warnings": []}`
- Category operations: `map[string]string` with per-service results

**Recommended structure:**
```go
// ActionResponse represents a consistent structure for action endpoints
type ActionResponse struct {
    Status   string                 `json:"status"`           // Required: action result (started/stopped/applied/resolved)
    Message  string                 `json:"message,omitempty"` // Optional: human-readable description
    Output   string                 `json:"output,omitempty"`  // Optional: command output (apply operations)
    Warnings []string               `json:"warnings,omitempty"` // Optional: non-fatal issues
    Metadata map[string]interface{} `json:"metadata,omitempty"` // Optional: operation-specific data
}
```

**Why this structure:**
1. **Status** - Required field consistent across all actions
2. **Message** - Optional human-readable context (e.g., "Stack disabled — enable it first")
3. **Output** - Docker Compose command output for apply operations
4. **Warnings** - Lock validation warnings, wire resolution warnings
5. **Metadata** - Extensible for operation-specific fields (resolved count, affected containers, etc.)

**Migration approach:**
- Wrap simple responses: `{"status": "started"}` → same (ActionResponse with only status)
- Preserve rich data: apply's `output` and `lock_warnings` map to ActionResponse fields
- Category operations: return ActionResponse with metadata containing per-service results

### Pattern 2: Swaggo Annotation Style

**What:** Comment-based API documentation using swaggo annotations

**When to use:** Above every HTTP handler function and in main.go

**Example for main.go:**
```go
// @title           DevArch API
// @version         1.0
// @description     Local microservices development environment API
// @termsOfService  http://swagger.io/terms/

// @contact.name   API Support
// @contact.url    http://www.devarch.local/support
// @contact.email  support@devarch.local

// @license.name  MIT
// @license.url   https://opensource.org/licenses/MIT

// @host      localhost:8550
// @BasePath  /api/v1

// @securityDefinitions.apikey ApiKeyAuth
// @in header
// @name X-API-Key
// @description API key for authentication

func main() {
    // ...
}
```

**Example for handler:**
```go
// Start godoc
// @Summary      Start a stack
// @Description  Start all instances in a stack (requires stack to be enabled)
// @Tags         stacks
// @Accept       json
// @Produce      json
// @Param        name   path      string  true  "Stack name"
// @Success      200    {object}  respond.SuccessEnvelope{data=respond.ActionResponse}
// @Failure      404    {object}  respond.ErrorEnvelope
// @Failure      409    {object}  respond.ErrorEnvelope
// @Failure      500    {object}  respond.ErrorEnvelope
// @Router       /stacks/{name}/start [post]
// @Security     ApiKeyAuth
func (h *StackHandler) Start(w http.ResponseWriter, r *http.Request) {
    // ...
}
```

**Key annotation rules:**
1. `@Summary` - One line description (imperative: "Start a stack" not "Starts a stack")
2. `@Description` - Additional context, constraints, requirements
3. `@Tags` - Logical grouping (stacks, instances, services, projects, networks)
4. `@Accept`/`@Produce` - Content types (always `json` except ExportStack which is `x-yaml`)
5. `@Param` - Path params, query params, body params with type and description
6. `@Success` - HTTP code + response type (use nested envelope syntax)
7. `@Failure` - All possible error codes with ErrorEnvelope type
8. `@Router` - Exact route path + HTTP method (must match routes.go)
9. `@Security` - Authentication requirement (ApiKeyAuth for all /api/v1 routes)

**Nested envelope syntax:**
```go
// For success responses wrapped in envelope
// @Success 200 {object} respond.SuccessEnvelope{data=respond.ActionResponse}
// @Success 200 {object} respond.SuccessEnvelope{data=[]models.Stack}
// @Success 200 {object} respond.SuccessEnvelope{data=models.Service}

// For error responses
// @Failure 400 {object} respond.ErrorEnvelope
// @Failure 404 {object} respond.ErrorEnvelope
```

### Pattern 3: CI Validation Workflow

**What:** Automated detection of breaking API changes in CI pipeline

**When to use:** On every PR that modifies handler code

**GitHub Actions workflow:**
```yaml
name: OpenAPI Validation

on:
  pull_request:
    paths:
      - 'api/internal/api/**'
      - 'api/docs/**'

jobs:
  openapi-diff:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout PR branch
        uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Set up Go
        uses: actions/setup-go@v5
        with:
          go-version: '1.22'

      - name: Install swag
        run: go install github.com/swaggo/swag/cmd/swag@latest

      - name: Generate current spec
        working-directory: api
        run: swag init -g cmd/server/main.go

      - name: Checkout base branch
        run: git checkout ${{ github.base_ref }}

      - name: Generate base spec
        working-directory: api
        run: |
          swag init -g cmd/server/main.go
          cp docs/swagger.yaml /tmp/base-swagger.yaml

      - name: Checkout PR branch
        run: git checkout ${{ github.head_ref }}

      - name: Install oasdiff
        run: go install github.com/oasdiff/oasdiff@latest

      - name: Check for breaking changes
        run: |
          oasdiff breaking /tmp/base-swagger.yaml api/docs/swagger.yaml \
            --fail-on-diff \
            --format githubactions

      - name: Generate diff report
        if: failure()
        run: |
          oasdiff changelog /tmp/base-swagger.yaml api/docs/swagger.yaml \
            --format markdown > /tmp/api-changes.md
          cat /tmp/api-changes.md >> $GITHUB_STEP_SUMMARY
```

**CI integration points:**
1. **Trigger:** Run on PR changes to `api/internal/api/**` or `api/docs/**`
2. **Generate:** Run `swag init` on both base and PR branches
3. **Compare:** Use `oasdiff breaking` with `--fail-on-diff` flag
4. **Report:** Output breaking changes in GitHub Actions format for PR annotations
5. **Document:** Generate markdown changelog for human review

**Breaking changes detected by oasdiff:**
- Removed endpoints
- Removed required parameters
- Added required parameters
- Changed parameter types
- Removed response fields
- Changed response types
- Changed error codes

### Anti-Patterns to Avoid

- **Incomplete annotations:** Missing `@Router`, `@Security`, or `@Failure` tags leads to incomplete spec
- **Type mismatch:** Annotation says `{object} models.Stack` but handler returns `map[string]interface{}` causes drift
- **Manual spec editing:** Never edit `docs/swagger.yaml` manually—always regenerate from annotations
- **Skipping CI validation:** Merging undocumented endpoint changes breaks API contracts
- **Inconsistent tagging:** Using both `stack` and `stacks` tags fragments documentation

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| OpenAPI spec generation | Custom reflection-based generator | swaggo/swag | Handles edge cases (nested types, pointers, enums, custom tags), maintained by community, 18k+ stars |
| Breaking change detection | Manual spec comparison | oasdiff | Comprehensive rule engine, handles semantic versioning, outputs multiple formats, battle-tested |
| Swagger UI hosting | Custom HTML/JS serving | swaggo/http-swagger | Pre-configured assets, handles CORS, supports customization, zero configuration |
| Response consistency | Ad-hoc map responses | Typed structs with optional fields | Type safety, OpenAPI generation, IDE autocomplete, refactoring support |

**Key insight:** OpenAPI tooling has evolved significantly. Hand-rolling spec generation or diff detection means reimplementing complex logic for JSON Schema generation, reference resolution, and semantic versioning rules. Mature tools handle these edge cases.

## Common Pitfalls

### Pitfall 1: Swag Annotation Drift

**What goes wrong:** Handler logic changes but annotations aren't updated, causing spec to diverge from actual behavior

**Why it happens:** Annotations are comments—compilers don't enforce correctness

**How to avoid:**
1. Make `swag init` part of pre-commit hooks
2. Run `swag fmt` to standardize comment formatting
3. Include generated `docs/` directory in version control
4. CI validation catches drift when base spec differs from PR spec

**Warning signs:**
- API client receives unexpected 404 (endpoint moved but spec not updated)
- Required parameter missing from spec (annotation omitted `@Param`)
- Response fields don't match spec (handler returns extra fields)

### Pitfall 2: Envelope Wrapping Confusion

**What goes wrong:** Forgetting that Phase 19 wraps all responses in `{data: ...}` or `{error: ...}` envelopes, causing incorrect OpenAPI schemas

**Why it happens:** Annotations reference the inner type but actual JSON response has outer envelope

**How to avoid:**
Use nested envelope syntax in annotations:
```go
// WRONG: Spec shows ActionResponse at root
// @Success 200 {object} respond.ActionResponse

// CORRECT: Spec shows envelope with nested data
// @Success 200 {object} respond.SuccessEnvelope{data=respond.ActionResponse}
```

**Warning signs:**
- API client expects `{"status": "started"}` but receives `{"data": {"status": "started"}}`
- Error responses missing `error.code` and `error.message` structure

### Pitfall 3: Chi Route Parameter Mismatch

**What goes wrong:** `@Router` annotation path doesn't match `routes.go` definition, causing "endpoint not found" errors

**Why it happens:** Chi uses `{param}` syntax but annotations might use `:param` or path/to/route structure differs

**How to avoid:**
1. Copy exact path from `routes.go` (e.g., `/stacks/{name}/instances/{instance}/start`)
2. Use chi's parameter syntax `{name}` not gin's `:name`
3. Cross-reference routes.go when writing annotations
4. Test generated spec by importing into Postman/Insomnia

**Warning signs:**
- Swagger UI shows route as `/stacks/:name/start` instead of `/stacks/{name}/start`
- Generated clients use wrong parameter names
- OpenAPI path matching fails in API gateways

### Pitfall 4: Auth Exemption Documentation

**What goes wrong:** Auth validation endpoints (`/api/v1/auth/validate`) are documented with `@Security ApiKeyAuth` despite being exempt from middleware

**Why it happens:** `/api/v1/auth/validate` is registered before the `r.Route("/api/v1", ...)` group that applies auth middleware

**How to avoid:**
1. Check routes.go to see if endpoint is inside auth-protected route group
2. Omit `@Security` annotation for endpoints registered outside `r.Route("/api/v1", ...)`
3. Add annotation comment explaining why auth isn't required: `// No @Security tag - this endpoint validates the API key itself`

**Warning signs:**
- Spec shows `/auth/validate` requires X-API-Key header in request
- Circular dependency documented (need key to validate key)
- API clients attempt to send auth header to validation endpoint

### Pitfall 5: Action Consistency with Category Operations

**What goes wrong:** Category start/stop endpoints return `map[string]string` with per-service results, breaking ActionResponse pattern

**Why it happens:** Categories operate on multiple services, needing per-service status

**How to avoid:**
Use ActionResponse with metadata field:
```go
type CategoryActionResponse struct {
    Status   string            `json:"status"`   // "completed"
    Services map[string]string `json:"services"` // Per-service results
}

respond.JSON(w, r, http.StatusOK, CategoryActionResponse{
    Status:   "completed",
    Services: results, // {"nginx": "started", "postgres": "error: ..."}
})
```

**Warning signs:**
- Category operations return different structure than stack/instance operations
- OpenAPI spec has two different action response schemas
- API clients need separate handling for category vs stack operations

### Action Response Standardization

```go
// api/internal/api/respond/types.go
package respond

// ActionResponse provides consistent structure for action endpoints
type ActionResponse struct {
    Status   string                 `json:"status"`           // Required: started/stopped/applied/resolved
    Message  string                 `json:"message,omitempty"` // Optional: human-readable message
    Output   string                 `json:"output,omitempty"`  // Optional: command output
    Warnings []string               `json:"warnings,omitempty"` // Optional: non-fatal issues
    Metadata map[string]interface{} `json:"metadata,omitempty"` // Optional: operation-specific data
}

// Action returns an action response wrapped in success envelope
func Action(w http.ResponseWriter, r *http.Request, statusCode int, status string, opts ...func(*ActionResponse)) {
    resp := &ActionResponse{Status: status}
    for _, opt := range opts {
        opt(resp)
    }
    JSON(w, r, statusCode, resp)
}

// WithMessage adds a message to action response
func WithMessage(msg string) func(*ActionResponse) {
    return func(r *ActionResponse) { r.Message = msg }
}

// WithOutput adds output to action response
func WithOutput(output string) func(*ActionResponse) {
    return func(r *ActionResponse) { r.Output = output }
}

// WithWarnings adds warnings to action response
func WithWarnings(warnings []string) func(*ActionResponse) {
    return func(r *ActionResponse) { r.Warnings = warnings }
}

// WithMetadata adds metadata to action response
func WithMetadata(key string, value interface{}) func(*ActionResponse) {
    return func(r *ActionResponse) {
        if r.Metadata == nil {
            r.Metadata = make(map[string]interface{})
        }
        r.Metadata[key] = value
    }
}
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Swagger 2.0 | OpenAPI 3.0/3.1 | 2017 (OpenAPI 3.0), 2021 (3.1) | Better JSON Schema support, webhooks, nullable handling, anyOf/oneOf |
| Manual YAML specs | Code-first with annotations | 2018+ (swaggo created) | Reduced drift, type safety, single source of truth |
| swagger-diff (unmaintained) | oasdiff, openapi-changes | 2023+ | Active maintenance, GitHub Actions support, configurable rules |
| Spec-first generation | Code-first or hybrid | 2020+ | Faster iteration for established APIs, spec-first still preferred for greenfield |
| gin framework focus | Multi-framework support | 2021+ | swaggo now supports chi, fiber, echo, standard net/http |

**Deprecated/outdated:**
- **Swagger 2.0:** Use OpenAPI 3.0. Swaggo generates 3.0 by default but can downgrade if needed
- **go-swagger:** Large, complex, generates code from spec. Overkill for annotation-based docs. Use swaggo for code-first
- **swagger-diff:** Archived project, last updated 2019. Use oasdiff or openapi-changes instead
- **Inline spec in routes.go:** Couples routing logic with documentation. Use annotations above handlers

## Open Questions

1. **Should we version the API?**
   - What we know: Currently using `/api/v1` base path, no version in spec
   - What's unclear: Whether Phase 20 should introduce semantic versioning in spec `version` field
   - Recommendation: Use `1.0` as initial version, document breaking vs non-breaking in changelog, consider major version bump (2.0) if Phase 20 ActionResponse changes break existing clients

2. **How to handle YAML export endpoint?**
   - What we know: `/stacks/{name}/export` returns `application/x-yaml`, exempt from JSON envelope (Phase 19-02 decision)
   - What's unclear: Whether swaggo annotation should show `@Produce yaml` or `@Produce x-yaml`
   - Recommendation: Use `@Produce application/x-yaml` and test spec generation. May need `swaggerignore:"true"` tag if envelope type causes issues

3. **Should we commit generated docs/ directory?**
   - What we know: `swag init` generates `docs/docs.go`, `swagger.json`, `swagger.yaml`
   - What's unclear: Trade-off between version control bloat vs ensuring spec is always available
   - Recommendation: Commit `docs/` directory—enables CI diff validation without regeneration step, provides historical record of API evolution, ensures spec matches code in every commit

4. **How granular should @Tags be?**
   - What we know: Tags group endpoints in Swagger UI sidebar (e.g., `stacks`, `instances`, `services`)
   - What's unclear: Whether to use single tags (`stacks`) or compound tags (`stacks,lifecycle` and `stacks,wiring`)
   - Recommendation: Single tags per resource (`stacks`, `instances`, `networks`). Compound tags fragment UI. Use `@Summary` and `@Description` for operation-specific context
