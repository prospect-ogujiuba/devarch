  <files>api/cmd/verify-boundary/main.go</files>
  <action>
    Create `api/cmd/verify-boundary/main.go` — a standalone CLI tool that tests import boundary behavior against a running API server.

    **Flags:**
    - `--api-url` (default: `http://localhost:8550`) — API server base URL
    - `--api-key` (default: read from `DEVARCH_API_KEY` env or empty) — API key for auth
    - `--stack-name` (default: `boundary-test`) — stack name for import test
    - `--json` — machine-readable JSON output

    **Synthetic YAML generation:**
    Create a function `generateSyntheticYAML(targetBytes int64) io.Reader` that produces a valid devarch.yml-format YAML stream of approximately `targetBytes` size.

    Strategy: Generate a base stack definition with many service instances. Each instance has a unique name like `boundary-svc-{N}` and realistic-looking config (image, ports, env vars, volumes). Duplicate services with incrementing names until the total size reaches the target. Use `io.Pipe()` or `bytes.Buffer` — for 200MB+ payloads, use a streaming approach with `io.Pipe()` to avoid holding the entire payload in memory.

    The YAML structure should match the devarch.yml export format:
    ```yaml
    name: boundary-test
    services:
      - name: boundary-svc-0001
        template: nginx
        overrides:
          environment:
            PADDING_VAR_1: "<long value to increase size>"
            ...
    ```

    Actually, since the import handler reads the YAML directly, generate a valid stack import YAML. Look at `api/internal/export/types.go` for the exact format.

    **Test execution:**

    1. **Pre-test setup:** Create a test stack via `POST /api/v1/stacks` with name from `--stack-name`. If it already exists, delete and recreate.

    2. **Test 1 — 200MB payload (VALD-03):**
       - Generate ~200MB synthetic YAML
       - POST as multipart/form-data to `/api/v1/stacks/{name}/import`
       - Assert HTTP 200 response
       - Print PASS/FAIL with payload size and response status

    3. **Test 2 — 300MB payload (VALD-04):**
       - Generate ~300MB synthetic YAML
       - POST as multipart/form-data to `/api/v1/stacks/{name}/import`
       - Assert HTTP 413 response
       - Parse JSON response body
       - Assert error message contains "exceeds" and "256MB"
       - Assert max_bytes field = 268435456 (256 << 20)
       - Assert received_bytes field is present
       - Print PASS/FAIL with details

    4. **Cleanup:** Delete the test stack.

    **Exit code:** 0 if both tests pass, 1 if any fails.

    **JSON output format (with --json):**
    ```json
    {
      "tests": [
        {
          "name": "200MB import succeeds",
          "pass": true,
          "payload_bytes": 209715200,
          "response_status": 200,
          "duration_ms": 1234
        },
        {
          "name": "300MB import rejected",
          "pass": true,
          "payload_bytes": 314572800,
          "response_status": 413,
          "response_error": "Import payload exceeds 256MB limit",
          "duration_ms": 567
        }
      ],
      "summary": { "passed": 2, "failed": 0, "total": 2 }
    }
    ```

    **Important implementation details:**
    - Use `mime/multipart` writer to create proper multipart form data
    - Use `io.Pipe()` for streaming — don't buffer 200MB+ in memory
    - Set `Content-Type` to `multipart/form-data; boundary=...`
    - Add `X-API-Key` header if api-key is provided
    - For the 200MB test, the YAML content doesn't need to be perfectly valid compose — it just needs to not crash the parser. Use large env var values as padding.
    - For the 300MB test, the rejection should happen at the middleware/handler level before full parsing, so content validity is irrelevant.
    - Handle connection resets gracefully (server may close connection on oversized payloads).
