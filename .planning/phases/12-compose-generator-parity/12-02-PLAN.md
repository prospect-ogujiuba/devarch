  <action>
  Create `api/cmd/verify-parity/main.go` — a CLI tool that:

  **1. Accepts flags:**
  - `--db` / `DATABASE_URL` — database connection string
  - `--compose-dir` — path to legacy compose directory (same as import cmd)
  - `--config-dir` — path to config directory (for config file import + materialization)
  - `--project-root` — for resolving relative paths in generator
  - `--service` — optional, verify single service by name (useful for debugging)
  - `--verbose` — print details for passing services too

  **2. Workflow:**
  a. Connect to DB, run fresh import: create Importer, call ImportAll(), ImportAllConfigFiles(), ResolveConfigMountLinks() — ensures DB reflects current compose files.
  b. For each service in DB (or just --service if specified):
     - Load the original compose file for this service (find by category + name in compose-dir)
     - Parse original with compose.ParseCompose() or compose.ParseFile() to get ParsedService
     - Create Generator, call MaterializeConfigFiles() then Generate() to get generated YAML
     - Parse generated YAML with same parser to get another ParsedService
     - Compare key fields between original and generated ParsedService

  **3. Fields to compare (per locked decision — matches Phase 15 list):**
  - Ports: host_ip, host_port, container_port, protocol (order-independent)
  - Volumes: source, target, read_only (order-independent, after config mounts removed from original)
  - EnvVars: key, value (order-independent)
  - EnvFiles: paths (order-dependent via sort_order)
  - Dependencies: service names (order-independent)
  - Healthcheck: test, interval, timeout, retries, start_period
  - Labels: key, value (order-independent)
  - Networks: network names (order-independent)
  - Config mounts: check that original config mounts appear in generated volumes (they're merged)
  - Image: name:tag
  - ContainerName / container_name
  - Restart policy
  - Command
  - User

  **4. Comparison approach:**
  For order-independent fields, sort both slices and compare. For maps, compare key sets and values. For optional fields (healthcheck, command, user), handle nil/empty.

  **5. Output format:**
  ```
  PASS  nginx (flowstate)
  PASS  php (flowstate)
  FAIL  redis (flowstate)
    - env_files: expected [./redis.env] got []
    - networks: expected [microservices-net app-net] got [microservices-net]

  Summary: 170/173 passed, 3 failed
  ```

  For each mismatch, print the field name, expected value (from original), and got value (from generated).

  **6. Exit code:** 0 if all pass, 1 if any fail.

  **Important notes:**
  - The parser's ParseCompose returns map[string]ParsedService keyed by service name. The generator's Generate() returns YAML bytes. Parse the generated YAML back through the same parser.
  - Some fields won't match exactly: the generator resolves relative paths (e.g., `../../apps/foo` becomes absolute). For volume source comparison, either compare only targets + read_only, or normalize both paths before comparing. The simplest approach: compare targets and read_only (source paths undergo resolution). Document any intentional relaxations.
  - Config mounts are in original's ConfigMounts but in generated's Volumes. The comparison needs to account for this: check that for each original ConfigMount, a matching volume entry exists in generated output.
